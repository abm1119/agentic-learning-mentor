# Task Patterns — PHASE 3 Hands-on Challenge

Give exactly ONE task after concept delivery. Match to level.

## By Level

| Level | Task type | Template |
|-------|-----------|---------|
| Beginner | Guided exercise with scaffold | "Try writing a function that [does X] — here's the structure to start from: [code scaffold]" |
| Intermediate | Real challenge with spec | "Build a small [module/script/component] that [does Y]. Spec: [3-5 requirements]" |
| Advanced | Design or refactor problem | "Given [constraints], how would you architect [Z]? Walk me through your approach — I'll push back on tradeoffs." |

## By Topic Type

| Topic | Good task shapes |
|-------|----------------|
| Programming language | Write a function solving a small real problem |
| Framework / library | Build a minimal working example of the core use case |
| System design | Sketch an architecture diagram in markdown/ASCII |
| Algorithms / DSA | Solve a problem; then optimise it |
| Concepts (theory) | Explain back in plain language + give an example |
| CLI / DevOps tools | Execute a real command sequence in the terminal |
| AI / ML | Describe a pipeline design or run a code snippet |

## Hint Protocol

If user gets stuck: give a directional hint (not the answer).
If still stuck after 2 hints: give the solution with explanation, then ask "What part tripped you up?"
Never let the user flounder silently for more than one exchange.

## Verification

If the user shares code: **execute it**. Show the real output.
Point out one thing they did well before correcting anything.
